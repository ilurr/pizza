## 2026-02-25 — Frontend refactor & performance pass

### 1. Data & API architecture

- Centralized **all mock data** under `src/data/` and wired API services to use it:
  - Added: `coverageAreas.json`, `authUsers.json`, `drivers.json`, `driverStock.json`, `driverEarnings.json`, `driverTransactions.json`, `driverNearbyDrivers.json`, and `index.js` re-export.
  - `DriverApiService` now imports driver profiles, stock, earnings, transactions, and nearby drivers from `src/data` instead of inline arrays.
  - `LocationPickerModal` imports coverage areas from `coverageAreas.json`.
  - `utils/auth.ts` uses `authUsers.json` for static dev login users.
- Made **ProductService** a thin facade over the API layer:
  - `getPizzas()`, `getBeverages()`, `getMenu()` now call `api.products.*` and fall back to local JSON on failure.
  - `getOrders(userId = 'guest_user')` now calls `api.orders.getUserOrders(userId)` and falls back to JSON.
- Fixed `PaymentService` import in `services/api/index.js` to use `@/service/PaymentService.js`.

### 2. Order & driver flows via API

- **Customer orders:**
  - `MyOrders.vue` and `useActiveOrders.ts` now pass a **user-specific id** into `ProductService.getOrders(userId)`, which forwards to `api.orders.getUserOrders(userId)`.
  - Active-orders polling still filters by status (`waiting`, `preparing`, `on_delivery`) but now uses per-user data.
- **OrderNow (customer side):**
  - Replaced inline driver list with `api.drivers.getAvailableDrivers(location)`, mapping API driver objects into the card shape (distance, estimated time, avatar, etc.).
- **Driver stock / earnings / exchange:**
  - `DriverStock.vue` loads inventory from `api.drivers.getDriverStock('driver_001')` instead of a local array.
  - `DriverEarnings.vue` calls `api.drivers.getDriverEarnings('driver_001', period)` and updates earnings/transactions per selected period.
  - `DriverExchange.vue` loads nearby drivers from `api.drivers.getNearbyDrivers('driver_001')`, mapping `availableStock` into the UI stock structure.

### 3. Location picker & Leaflet

- `LocationPickerModal.vue` now:
  - Loads **Leaflet CSS/JS on demand** via `ensureLeafletLoaded()` only when the modal is shown.
  - Uses `coverageAreas.json` for delivery coverage polygons.
  - Calls `map.invalidateSize()` after initialization to fix the blank-map-in-modal issue.
  - Keeps pin and address logic intact, with coverage checks against the configured polygons.

### 4. Performance optimizations

- **Lazy-loaded heavy components** using `defineAsyncComponent`:
  - `CartModal`, `LocationPickerModal`, `ProductDetailModal` in `OrderNow`.
  - `ProductDetailModal` in `Menu` and `Landing`.
  - `OrderTrackingModal` in `OrderList`.
  - `PromoModal` in `CartModal`.
- **Active-orders polling (`useActiveOrders`)**:
  - Added document `visibilitychange` handling:
    - Stops polling when the tab is hidden.
    - Restarts polling (with an initial fetch) when the tab becomes visible and user is authenticated.
  - Guards against duplicate intervals.
- **Debounced order-store persistence**:
  - Added a small debounce wrapper for `localStorage.setItem` and wired it into Pinia’s `persist` config in `orderStore`.
  - Order state (`currentStep`, `userLocation`, `selectedDriver`, etc.) still persists, but writes are batched (~300 ms).

### 5. UX & images

- Improved image loading:
  - `CartModal` cart item images now use `loading=\"lazy\"`.
  - `OrderList` empty-state illustration also uses `loading=\"lazy\"`.
  - Existing `ImageWithSkeleton` usage remains for product and avatar images.
- No visual regressions expected; splash screen, login, and topbar still reference existing assets as before.

### 6. Env & configuration

- Added `.env.example` documenting expected environment variables:
  - `VITE_DATA_SOURCE`, `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`, `VITE_USE_MOCK_API`, `VITE_API_BASE_URL`, `VITE_XENDIT_WEBHOOK_SECRET`.
  - Optional: `VITE_OPENROUTESERVICE_API_KEY` (used in `useLeaflet`).
- `ApiClient` and `PaymentService` continue to read from `import.meta.env` with sane defaults; this example file serves as the source of truth for configuring dev/prod.
